CS 4880 P0 — In-Depth Code Walkthrough with Requirement References

Callouts labeled REQ cite the exact spec or suggestion document that drove each implementation decision. Callouts labeled +10% mark the extra-credit validation feature.

High-Level Goal

The program reads whitespace-delimited words, counts how many times each word appears, then organizes them into a BST keyed by frequency (not alphabetically). It writes three output files: the BST in preorder, inorder, and postorder traversal.

REQ — P0 Specification (p.1): "Write a program to build a tree for a file input, and then print the tree using three different traversals." The three traversal output files are the primary deliverable, worth 60% of the grade.

File & Folder Structure

REQ — Structural Requirements (P0.pdf p.3): "The main function should be in the file P0.c … any helpers helping it processing the argument or generate errors in the same file. Then, you should have functions buildTree() printInorder() printPreorder() printPostorder() implemented in the same file tree.c/tree.cpp with tree.h as needed. If you have a type definition for the node, it should be in a separate header file node.h."
This is why the project is split into exactly three source units: P0.c (main + argument helpers), tree.c / tree.h (tree functions), and node.h (node type). Mixing them would lose architectural standards points (10% of grade).

Data Structures

node.h — The BST Node

typedef struct Node {
    int   frequency;   // BST key
    char **strings;    // dynamic array of words sharing this frequency
    int   str_count;
    int   str_cap;
    Node *left, *right;
} Node;
REQ — P0 Specification (p.2): "Data repeated the same number of times is interpreted as equivalent data from the ordering perspective (thus falling in the same node)." and "A node retains each repeated datum just once."
This is why a single Node holds a list of strings rather than a single string — multiple words can share one frequency and therefore one node.
REQ — Structural Requirements (P0.pdf p.3): "If you have a type definition for the node, it should be in a separate header file node.h."
The struct lives in node.h, not inline in tree.c or P0.c.

tree.c (internal) — StringEntry Registry

typedef struct {
    char *token;
    int   frequency;
} StringEntry;
REQ — P0 Suggestions (p.1–2): "The tree is built using the frequency of a given string in the file. Because the frequency is unknown when seeing a string, the program should make two passes over the input. The first pass will collect the strings along with frequencies (use a container, a dictionary, etc.). The second pass will read the container and construct the tree without rearranging."
StringEntry is that intermediate container — it stores each unique token and its final count so that Pass 2 can build the BST with stable, finalized frequencies.

main() in P0.c — Three Top-Level Tasks

REQ — P0 Suggestions (p.1): "Using top-down decomposition you have 3 tasks in main: (1) Process command arguments … (2) Build the tree (3) Traverse the tree three different ways, generating outputs."
The three sections of main map directly onto these three tasks. The suggestions doc even provides sample pseudocode showing root = buildTree(file) followed by the three print calls.

buildTree() in tree.c — The Two-Pass Core

REQ — P0 Suggestions (p.1–2): "Because the frequency is unknown when seeing a string, the program should make two passes over the input. The first pass will collect the strings along with frequencies. The second pass will read the container and construct the tree without rearranging."
This is the architectural reason the function exists at all — if frequencies were known upfront, a single-pass insert would work. They aren't, so two passes are mandatory.

Pass 1 — Build the Registry (tree.c lines 103–128)

while (fscanf(in, "%1023s", buf) == 1) {
    if (!is_alphanumeric(buf)) { /* warn and skip */ }
    // linear search registry for existing token
    // if found: increment frequency
    // if not found: grow registry and add new StringEntry
}
REQ — Requirements (P0.pdf p.2): "Assume the input data is all strings (printable ASCII) separated with standard WS (white space) separators (space, tab, new line)."
fscanf(in, "%s", ...) is the natural C idiom for this — it skips all whitespace automatically and reads one whitespace-delimited token, matching the spec exactly.
REQ — Requirements (P0.pdf p.2): "Assume you do not know the size of the input file."
This is why the registry is a dynamic array starting at 16 slots and doubling on each reallocation — no fixed upper bound is assumed.
+10% Extra Credit — Validation (P0.pdf p.2): "The data is all strings made up of letters and digits only. You may assume clean data, +10% for validation — a warning message on any bad datum and then skipping that datum."
is_alphanumeric() checks every character with isalnum(). Non-alphanumeric tokens print a warning to stderr and are skipped entirely — they never enter the registry and never affect the BST. The unsigned char cast avoids undefined behavior on signed char values, which isalnum requires.

Pass 2 — Build the BST (tree.c lines 131–143)

for (i = 0; i < reg_count; i++) {
    Node *existing = bst_find(root, registry[i].frequency);
    if (existing)
        node_add_string(existing, registry[i].token);
    else {
        Node *new_node = make_node(registry[i].frequency, registry[i].token);
        root = bst_insert(root, new_node);
    }
    free(registry[i].token);
}
free(registry);
REQ — Requirements (P0.pdf p.2): "Tree is never balanced nor restructured other than growing new nodes as needed, and the tree is constructed using the first occurrence of each string."
The registry preserves first-appearance order (tokens are appended in the order they were first seen). Pass 2 iterates the registry in that order, so the BST is built using first-occurrence positions as required. There is no balancing or restructuring — once a node is inserted it never moves.
REQ — Requirements (P0.pdf p.2): "Data repeated the same number of times is interpreted as equivalent data from the ordering perspective (thus falling in the same node)."
bst_find detects an existing node with the same frequency. When found, the token is appended to that node's string list (node_add_string) rather than creating a new node — exactly the "same node" behavior the spec requires.

BST Ordering — Why Higher Frequency = Right

// bst_insert (tree.c lines 72-75):
if (new_node->frequency < root->frequency)
    root->left  = bst_insert(root->left,  new_node);
else
    root->right = bst_insert(root->right, new_node);
REQ — Requirements (P0.pdf p.2): "any datum repeated in the file a number of times is larger than another datum that was repeated a smaller number of times."
"Larger" maps to the right subtree in a standard BST. So higher frequency → right, lower frequency → left. This also means an inorder traversal naturally produces ascending frequency order.

Node Print Format (tree.c lines 81–90)

static void print_node(Node *n, int depth, FILE *out) {
    int i;
    for (i = 0; i < 2 * depth; i++)
        fputc(' ', out);
    fprintf(out, "%d:", n->frequency);
    for (i = 0; i < n->str_count; i++)
        fprintf(out, " %s", n->strings[i]);
    fputc('\n', out);
}
REQ — Printing in Traversal (P0.pdf p.3): "a node will print starting with indentation of 2x depth (root is at depth 0) then the node's number (the frequency for the node's data) followed by the character ':' and WS followed by the list of strings from the node separated by WS (retaining the original ordering as according to appearance in the input file)."
Every part of this format is directly mandated: the 2 * depth leading spaces, the frequency integer, the literal ':', the space separator, and the strings in first-appearance order. The P0 Suggestions doc also shows a printf("%*c%d:…", level*2, ' ', level, …) sketch confirming the same indent formula.

The Three Traversal Functions (tree.c lines 146–168)

REQ — Traversals (P0.pdf p.3): "Traversals taken from the 3130 textbook: Preorder: process root / process children left to right. Inorder: process left child / process root / process right child. Postorder: process children left to right / process root."
The three functions implement exactly these definitions. Depth is passed as depth + 1 on every recursive call so each child level gets 2 more spaces of indentation — but depth represents structural position in the BST, not visit order, so it is always root_depth + 1 for both children regardless of traversal.
